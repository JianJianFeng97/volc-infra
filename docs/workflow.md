# GitOps Workflow

This document describes how the IaC GitOps workflow is structured for the
`JianJianFeng97/volc-infra` repository.

---

## 1. Overview

All infrastructure is managed with **Terraform** (`>= 1.6`) and targets
**Volcengine** (火山引擎) in the `cn-shanghai` region.

Changes follow the standard GitOps loop:

```
feature branch ──PR──> main
                   │
              CI pipeline
         (fmt / init / validate / plan)
                   │
              human review
                   │
             manual apply
         (local or CI with approval)
```

> **No automatic `apply`.**  Every `apply` is a deliberate, human-initiated
> step to prevent unintended changes in production.

---

## 2. Environments

| Environment | Directory            | Name prefix | Tag `Environment` |
|-------------|----------------------|-------------|-------------------|
| Development | `infra/envs/dev/`    | `dev-`      | `dev`             |
| Production  | `infra/envs/prod/`   | `prod-`     | `prod`            |

Both environments live in the **same Volcengine account** and are distinguished
exclusively by:

- Resource **name prefix** (`dev-` / `prod-`).
- Resource **tags** (see §3).

---

## 3. Standard Tags

Every resource must carry the following tags, produced by
`infra/modules/tags`:

| Tag key       | Value                          |
|---------------|--------------------------------|
| `Environment` | `dev` or `prod`                |
| `ManagedBy`   | `terraform`                    |
| `Region`      | `cn-shanghai`                  |
| `Repo`        | `JianJianFeng97/volc-infra`    |

Additional tags can be passed via the `extra_tags` variable.

---

## 4. Local Apply

```bash
# 1. Export short-lived credentials (never store in files).
export VOLCENGINE_ACCESS_KEY="<ak>"
export VOLCENGINE_SECRET_KEY="<sk>"

# 2. Initialise (with a real backend or locally).
cd infra/envs/dev
terraform init        # or: terraform init -backend=false for local state

# 3. Review the plan.
terraform plan \
  -var="volcengine_access_key=$VOLCENGINE_ACCESS_KEY" \
  -var="volcengine_secret_key=$VOLCENGINE_SECRET_KEY"

# 4. Apply after review.
terraform apply \
  -var="volcengine_access_key=$VOLCENGINE_ACCESS_KEY" \
  -var="volcengine_secret_key=$VOLCENGINE_SECRET_KEY"
```

---

## 5. Credential Management

- **Never** commit AK/SK (Access Key / Secret Key) to the repository.
- `*.tfvars` files are listed in `.gitignore` to prevent accidental leakage.
  Only `terraform.tfvars` (which contains no secrets) is tracked.
- For production, prefer **short-lived credentials** generated by Volcengine
  IAM role assumption, or an **OIDC-based** federation if available.
- For GitHub Actions, store credentials as
  [GitHub Actions Secrets](https://docs.github.com/en/actions/security-guides/encrypted-secrets):
  - `VOLCENGINE_ACCESS_KEY`
  - `VOLCENGINE_SECRET_KEY`

---

## 6. State Management

- The default configuration uses a **local** backend (state lives in
  `.terraform/` which is gitignored).
- For team usage, switch to a remote backend backed by
  **Volcengine TOS** (S3-compatible).  See `backend.tf` in each environment
  directory for a commented-out example.
- **Never commit `*.tfstate*` files** – they may contain secrets and large
  diffs that make PRs unreadable.

---

## 7. CI Pipeline

The CI pipeline (`.github/workflows/terraform-ci.yml`) runs on every PR
targeting `main` that touches `infra/**`:

| Step                  | Requires credentials | Fails PR on error? |
|-----------------------|----------------------|--------------------|
| `terraform fmt -check`| No                   | Yes                |
| `terraform init -backend=false` | No         | Yes                |
| `terraform validate`  | No                   | Yes                |
| `terraform plan`      | **Yes**              | No (warning only)  |

`terraform plan` is marked `continue-on-error: true`.  Once you add
`VOLCENGINE_ACCESS_KEY` / `VOLCENGINE_SECRET_KEY` as repository secrets,
the plan will succeed and you can optionally change the step to be required.
